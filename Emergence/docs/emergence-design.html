Emergence

An interactive visionOS simulation of the hydrogen atom's quantum probability density, designed for Apple Vision Pro.

visionOS 2+ Swift / SwiftUI RealityKit ARKit ArtCenter Capstone 2026 Ben Kase

Vision & Concept

Emergence places the user inside the hydrogen atom. The atom's electron probability density — the real-world quantum cloud described by the Schrödinger equation — is rendered as a million-point particle cloud that hovers in mixed reality or fills a pitch-black quantum space.

The name reflects a core insight: complex, ordered structure emerges from simple quantum rules. Each orbital shape — the dumbbell of a p-orbital, the cloverleaf of a d-orbital — is not designed but derived directly from wavefunction mathematics. The user manipulates and fires photons at the atom, watching it absorb energy, jump to excited states, and spontaneously decay back toward the ground state with physically-accurate selection rules and timescales.

Design principles that guided every decision

  • Physically accurate first. Energy levels, selection rules, wavelengths, and lifetimes use CODATA-2018 constants. The display is a faithful projection of quantum mechanics, not an artistic approximation.
  • Match the platform's native feel. Every gesture uses the same pipeline visionOS itself uses — system-predicted hand tracking, not custom thresholds. The app should feel indistinguishable from a first-party Apple app in how it responds to the user's hands.
  • The science is the art. The fiery warm colormap, the cold lavender of transition superpositions, the glowing photon spheres — all are derived from the physics. No decorative elements are added that don't encode real information.

The app is structured as three independent layers that communicate through AppModel (an @Observable singleton). The physics layer runs in a polling loop at ~125 Hz; rendering runs at the compositor's 90 Hz frame rate via TimelineView(.animation); the interaction layer is purely event-driven through SwiftUI gestures.

Architectural choice Combine was intentionally avoided. All async work uses Swift's structured concurrency (async/await, Task, actor). The physics loop is a cancellable Task sleeping 8 ms per iteration; the ARKit session uses async for over HandTrackingProvider.anchorUpdates.

Quantum Mechanics

What is being visualized

The electron in a hydrogen atom occupies a bound state characterized by three quantum numbers:

SymbolNameRangeMeaning
nPrincipal1 – 6Energy level; determines orbital size
ℓAngular momentum0 – n-1Orbital shape (s, p, d, f, g, h)
mMagnetic-ℓ – +ℓOrientation of angular momentum

The rendered cloud is the probability density |ψnℓm(r,θ,φ)|², where ψ is the real-valued spherical harmonic combination of the analytic wavefunction. Points are sampled proportional to this density, so denser regions of the cloud correspond to higher probability of finding the electron.

Energy levels (Bohr formula)

// E_n = -13.606 eV / n²
static func energyLevel(_ n: Int) -> Double {
    -rydbergEv / Double(n * n)
}

Selection rules (E1 / dipole transitions)

Only transitions satisfying Δℓ = ±1 and |Δm| ≤ 1 are allowed. This is enforced in both absorption matching and spontaneous emission targeting. The 2s orbital is metastable because no allowed E1 path to 1s exists (Δℓ would be 0, not ±1).

Time dilation

The actual excited-state lifetime for n=2 is ~1.6 ns. Scaled by the display slowdown factor (1.17 × 10⁸), a minimum dwell of 20 display-seconds is enforced so users can observe each orbital before the atom decays.

Wavefunction Math

All wavefunction evaluation is in WavefunctionMath.swift. The key function is psi2Real, which evaluates the real-valued probability density at a Cartesian point:

// Real spherical harmonic combination (no complex exponentials)
static func psi2Real(x: Double, y: Double, z: Double,
                       n: Int, l: Int, m: Int, a: Double) -> Double

The effective Bohr radius aEff is reduced for high-n orbitals so the outermost lobe stays within the 0.35 m display sphere:

private static func effectiveA(for n: Int) -> Double {
    let maxAllowed = displayRBox / Double(2 * n * n)
    return min(displayAWorld, maxAllowed)
}

Point sampling uses a 128³ grid CDF (cumulative distribution function) with Gaussian jitter (σ = 1.2 × grid step) applied per-sample. This avoids hard cube-face artifacts and produces a smooth, continuous-looking cloud.

Transitions & Photons

Photon firing

The user fires photons with SpatialTapGesture (a pinch-tap on the atom entity). A glowing sphere is spawned at the user's pinch position and travels to the atom's current position at 2 m/s. On arrival, AtomStateMachine.tryAbsorb checks for resonance within ±0.08 eV. If resonant, the atom jumps to the target state; if the photon energy exceeds the ionization threshold, the atom ionizes and recombines after a 2-second delay.

Transition animation (superposition rendering)

During a quantum jump, TransitionController advances a phase angle φ from 0→2π over ~1.5 seconds. The point cloud switches from the static 1M-point orbital render to a 500K-point superposition render:

// Superposition density: |ψ_A + ψ_B·e^(iφ)|²
let d = 0.5 * (pA*pA + pB*pB) + pA*pB*cosPh

Points in regions of constructive interference are rendered in the warm fire colormap; destructive interference regions use the cool indigo-lavender colormap, encoding the quantum phase structure of the two-state superposition.

Design decision The transition render uses a coarser 32³ grid (vs the static 128³) to keep per-frame rebuild under ~16 ms on the Vision Pro GPU. Resolution was traded for frame rate during the animation.

1M-Point Cloud Rendering

The probability cloud is rendered as exactly 1,000,000 point primitives using LowLevelMesh — RealityKit's lowest-overhead mesh API. Each vertex carries a 3D position and a packed BGRA color (4 bytes). The total buffer is 28 MB.

private struct PointVertex {
    var position: SIMD3<Float> = .zero
    var color: UInt32 = 0  // packed BGRA
}

Colormap

Point color encodes the local probability density, normalized to the orbital's maximum. The warm "fire" ramp (deep violet → red → orange → white) makes high-density regions (near antinodes) glow hottest:

Density rangeColor
0 – 20%Deep violet → dim purple
20 – 45%Purple → red
45 – 70%Red → orange
70 – 88%Orange → warm yellow-white
88 – 100%White-hot

Rebuild pipeline

When the quantum state changes, a rebuild is dispatched to a DispatchQueue.global(.userInitiated) background thread. After the 16 MB buffer copy, a Task.yield() allows hand-tracking updates to run before committing the LowLevelMesh.Part that makes the new cloud visible. This prevents a frame drop during the largest state changes.

ShaderGraph Material

PointCloudMaterial.usda is a RealityKit ShaderGraph material loaded from the RealityKitContent package. Its key properties:

  • Unlit surface shader — the cloud glows regardless of scene lighting; perfect for quantum "glow"
  • Vertex color pass-through — the packed per-vertex BGRA maps directly to surface color
  • Opacity 0.8 — slight transparency gives depth to overlapping point layers
  • Depth write enabled — points correctly occlude each other
  • Point topology — rendered as individual screen-space quads, not triangles

HandTracker

HandTracker.swift wraps ARKit.HandTrackingProvider and WorldTrackingProvider in a @MainActor @Observable class. It provides:

left.isPinchingBoolThumb-index distance < 15 mm

Pinch threshold

The threshold is 15 mm (reduced from an original 22 mm). The 22 mm value caused near-pinch false positives — fingers close together but not touching would trigger scale/rotate gestures. The tighter 15 mm threshold matches the physical reality of an actual pinch.

Role of HandTracker vs system gestures HandTracker is now used only for: (1) the two-hand midpoint translation during MagnifyGesture, (2) distinguishing hand-occluded from hand-open states during drag, and (3) coasting cancellation. All gesture detection has been migrated to system SwiftUI gestures.

One-Hand Drag

Why system gestures, not custom ARKit tracking

The original implementation used raw ARKit joint positions (HandTracker.activePinchPoint) for drag. This caused a critical UX problem: when the user's thumb occludes the index finger from the Vision Pro cameras during certain hand orientations, ARKit loses the pinch and the atom freezes. When tracking recovers, the atom snaps to the new position.

DragGesture(minimumDistance: 0).targetedToAnyEntity() uses the same predictive ML model that drives visionOS window dragging. The system extrapolates hand position through brief tracking gaps, producing a continuous stream with no freezes.

Predicted position extraction

// inputDevicePose3D is in entity-local (SwiftUI Y-down) space.
// value.convert moves it to RealityKit scene (Y-up) world space.
if let pose = value.inputDevicePose3D {
    let s = value.convert(pose.position, from: .local, to: .scene)
    pt = SIMD3<Float>(Float(s.x), Float(s.y), Float(s.z))
} else if let hp = handTracker.activePinchPoint {
    pt = hp  // Simulator fallback
}

Delta-drag formula

The atom's position is computed from an anchor rather than accumulated deltas. This prevents drift:

atomPos = dragAtomStart + (currentPinch − dragStartPinch) × dragScale

dragScale — angular amplification

The atom can be 1–2 meters away while the hand moves only centimeters. dragScale amplifies hand motion proportionally to the atom's distance from the head:

dragScale = min(atomDist / handDist, 1.3)

Capped at 1.3× to prevent runaway amplification for very distant atoms.

Open-palm guard

The system gesture can remain active briefly after a pinch releases. A guard distinguishes "hand occluded" (let the prediction continue) from "hand visibly open" (stop dragging):

// Hand visible + not pinching = genuinely open → end drag
if handTracker.isAnyHandTracked && !handTracker.activePinching {
    if isDraggingAtom { startCoastingIfFast() }
    isDraggingAtom = false
    return
}

Movement threshold

Drag doesn't commit until the pinch has moved 1.5 cm from its starting position. This prevents a tap from accidentally initiating a drag.

Two-Hand Scale & Rotate

System gesture pipeline

The original two-hand implementation used raw ARKit hand positions with a manual distance ratio for scale and cross-product for rotation. This was replaced with MagnifyGesture and RotateGesture3D, which use the same system ML pipeline as visionOS window resizing and rotating. The system handles pinch detection — no manual thresholds are involved.

GestureValue usedApplied as
MagnifyGesturevalue.magnification — cumulative scale factor from gesture startnewScale = anchorScale × magnification
RotateGesture3Dvalue.rotation.quaternion — cumulative Rotation3Dorientation = dq × anchorRot

Midpoint translation

During magnify, the atom's center tracks the midpoint between both hands. HandTracker provides the raw positions since no system gesture exposes absolute world-space hand positions:

let mid = (handTracker.left.pinchPoint + handTracker.right.pinchPoint) * 0.5
atomContainer?.position = anchorPos + (mid − anchorMid)

Coordinate axis correction for RotateGesture3D

RotateGesture3D.Value.rotation arrives in a coordinate frame that disagrees with RealityKit world space on two of three axes:

AxisGesture conventionRealityKitFix
Y (up)+Y up+Y upNone — correct
X (right)+X right, but rotation direction inverted+X rightNegate ix
Z (depth)+Z toward user−Z forwardNegate iz
let qd = value.rotation.quaternion  // simd_quatd
// Negate ix and iz; leave iy (Y/up axis is shared)
let dq = simd_quatf(ix: -Float(qd.imag.x), iy: Float(qd.imag.y),
                    iz: -Float(qd.imag.z), r: Float(qd.real))

This correction was arrived at empirically by testing one axis at a time: Y worked from the start; Z ("parallel to screen" rotation) was flipped first and required negating iz; X ("horizontal axis, left-to-right") was tested last and required negating ix.

Scale limits

The atom scale is clamped to [0.2×, 10×] of the initial 2× scale. The hit sphere inverse-scales to maintain constant world-space input target radius regardless of atom scale.

Photon Firing (Tap)

Photon firing uses SpatialTapGesture().targetedToAnyEntity(), which uses visionOS's ML-based contact detection — the same system that powers button taps throughout the OS. This replaces an earlier custom threshold approach (22 mm thumb-index distance) that fired too easily from near-pinches.

The photon origin is the user's last known pinch position, captured during drag (lastKnownPinchPos) and updated each drag frame. This ensures that even after fingers separate for the tap, the photon launches from the right location.

// .gesture runs with higher priority than .simultaneousGesture
.gesture(SpatialTapGesture().targetedToAnyEntity().onEnded { _ in
    guard photonFiringEnabled, !isDraggingAtom, !isTwoHandScaling else { return }
    firePhoton(from: lastKnownPinchPos ?? simulatorHandPos,
               ev: atomState.dialedPhotonEv)
})

Coasting (Momentum)

After a fast drag release, the atom coasts with exponential velocity decay, matching visionOS window behavior. The velocity is tracked during drag using an exponential moving average (EMA) to smooth out per-frame jitter:

// EMA: 80% new sample, 20% history — responsive but not noisy
atomVelocity = atomVelocity * 0.2 + rawVel * 0.8

On drag end, if the speed exceeds coastThreshold (0.25 m/s), coasting begins with speed capped at 2.5 m/s:

// Decay: v(t) = v₀ · e^(−momentumDecay · t)
// momentumDecay = 5.0 → half-life ≈ 140 ms
atomVelocity *= exp(-momentumDecay * dt)
Tuning history The initial decay constant was 9.0 (half-life ~77 ms), which felt too abrupt compared to visionOS windows. Reduced to 5.0 (~140 ms) to match the window glide feel. Coasting integrates position in the 8 ms polling loop, not the 90 Hz RealityKit update closure, to avoid writing @State inside the compositor callback.

Control Panel (ContentView)

The SwiftUI window provides full control over the simulation without entering the immersive space:

  • Current State — live orbital label, binding energy in eV, degeneracy count
  • Quantum State Picker — cascading pickers for n, ℓ, m with constraint enforcement (ℓ < n, |m| ≤ ℓ), Apply button bypasses photon absorption and pins the atom
  • Photon Energy — slider 1.5–14.0 eV with color swatch showing the photon's visible wavelength color
  • Snap to Resonance — scrollable row of buttons that snap the slider to each available transition energy
  • Immersion Mode — segmented control: Mixed Reality vs Quantum Realm
  • Photon Firing Toggle — disable taps to observe the atom without accidentally firing

Photon Energy Slider

The slider range (1.5–14.0 eV) covers:

  • All Lyman, Balmer, Paschen, Brackett, Pfund, and Humphreys series transitions (n=1–6)
  • The ionization threshold of the ground state (13.6 eV)

The color swatch converts the dialed eV to a wavelength via hc/E and uses the Dan Bruton wavelength-to-RGB approximation (with gamma 0.8 and edge rolloff). UV photons show dim violet; IR photons show dim red. A resonance indicator shows green (exact match ±0.08 eV) or yellow (ionizing).

Key Design Decisions

Decision: Use system gesture pipeline throughout All gesture detection was migrated from custom ARKit threshold comparisons to SwiftUI system gestures (DragGesture, MagnifyGesture, RotateGesture3D, SpatialTapGesture). This provides the same quality of tracking that visionOS itself uses for windows, with predictive extrapolation through occlusion gaps and ML-based contact detection. The app's interactions feel native because they are native.
Decision: Real spherical harmonics (not eigenstate densities) The real-valued combinations of ψnℓm are visualized rather than the complex eigenstates. This is the conventional choice in chemistry education (px, py, dxy, etc.) and produces the familiar dumbbell and cloverleaf shapes that match textbook orbital diagrams.
Decision: Importance sampling over uniform sampling Points are drawn from a CDF built over the 128³ density grid, not placed uniformly in space. This concentrates points where the probability density is high, making orbital structure legible even in regions with very different density scales (e.g., a 3d orbital has both a core region and outer lobes).
Decision: LowLevelMesh over ModelEntity LowLevelMesh with manual buffer management was chosen over ModelEntity because it allows zero-copy CPU→GPU updates of exactly the 28 MB vertex buffer without going through RealityKit's mesh builder pipeline. This was necessary to keep state-change rebuilds under 16 ms on device.
Decision: No Combine Combine was explicitly avoided throughout. All asynchronous work uses Swift's structured concurrency. The user specified this preference at the architectural level, and it produces cleaner cancellation semantics, especially for the rebuild tasks that must be cancelled when a new state change arrives before the previous rebuild finishes.
Decision: Delta-drag formula (not accumulated deltas) Atom position is computed as atomStart + (currentPinch − startPinch) × scale rather than accumulating frame-to-frame deltas. Accumulated deltas drift due to floating-point error and produce jitter when the velocity EMA is active. The anchor formula is stable regardless of frame rate.
Decision: No @State writes in the RealityKit update: closure The RealityView update: closure runs at compositor frame rate (90 Hz). Writing @State inside it triggers SwiftUI re-renders every frame, causing a cascade. All mutable state that needs to be read in update: is written in gesture handlers or the 8 ms polling loop, and only read inside the compositor callback.
